<objective>
Wire the D-12 two-key delivery gate for replays, **extended with explicit authorization-payload sourcing** (the third key). Replay defaults to no delivery; ALL THREE of the following must be true to deliver from a replay:

1. `factory-config.replay.allowDelivery: true` (config key)
2. `--allow-delivery` argv flag (per-invocation key)
3. `--authorization-file <path>` argv (operator-supplied authorization payload — fed to `reAuthorizeFromPayload`, NEVER re-minted)

The third key resolves the gap that failed parent runs lack `runs/<parentRunId>/delivery/authorization.json` on disk. Rather than silently skipping delivery for failed runs (which would make replay-delivery non-deterministic w.r.t. parent-run state), we require the operator to explicitly supply the authorization payload — same shape as a fresh `deliver` invocation, validated by the same boundary.

Each delivery-from-replay event audits to `admission-decisions.jsonl` under a new `gate: "replay-delivery"` value (Pitfall #5 — `GateName` is a closed enum at `packages/authority/src/admission-decision/base.ts:3-10`; this plan extends it).

Also pins REPLAY-14: the admission re-fire contract is unreachable in v1 (D-02 forbids editing admission-input artifacts) but the contract test exists so v2's editable-allowlist expansion is safe-by-default.

Purpose: replay must compose with Phase 2's authority kernel without inventing a side channel. This plan is the only place where replay can ever produce a real PR.
Output: three-key gate + audit + REPLAY-14 contract + positive-path delivery test.
</objective>
