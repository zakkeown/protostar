## Phase Boundary

Align package manifests + dependency-boundary enforcement with the source-level architecture so the 20-package monorepo reads as deliberate separation of concerns rather than accidental sprawl. Strict scope: package.json shape, tier classification, no-fs/no-net contract tests, tsconfig project-references alignment, AGENTS.md tier-table + forbidden-edge prose. **No source moves, no package merges, no package splits** — those belong to a separate phase if they prove necessary later. Wires `apps/factory-cli` for npm publish (`@protostar/factory-cli` with `bin`). Lands BEFORE Phase 11's headless-mode work, so any tier reclassification this phase makes constrains how Phase 11 wires CI runners.

## 1. Trigger & Scope

### Q-01 — Insertion trigger
**Decision:** Operator-observed package sprawl (20 packages) during Phase 10 dogfood work. Preventive hygiene, not bug-driven.
**Rationale:** No specific failure prompted 10.1; the felt-sense of an unreviewed, growing package surface justifies a deliberate alignment pass before Phase 11 multiplies it.
**Status:** Decided.

### Q-02 — Phase scope
**Decision:** **Strict + AGENTS.md prose.** Manifests, contract tests, tsconfig refs, AGENTS.md tier-table refresh, and explicit forbidden-edge documentation in AGENTS.md prose. **No source moves, no package merges/splits.**
**Rationale:** Strict reading honors the phase goal text; AGENTS.md prose update is cheap and locks the new tier classifications into the developer-facing contract. Restructuring is a separate, riskier phase that would block dogfood; defer.
**Note for planner:** AGENTS.md changes are an artifact of this phase, not a precursor — the prose update lands in the same plan as the corresponding code/manifest changes (e.g. tier-classification plan rewrites the AGENTS table; back-edge-audit plan adds the forbidden-edge documentation).
**Status:** Decided.

### Q-03 — Commit granularity
**Decision:** **Per-concern plans (~5–7).** Group by concern: tier classification + AGENTS table refresh; no-fs/no-net contracts; tsconfig refs alignment; manifest hygiene sweep; back-edge audit; workspace-wide conformance gate.
**Rationale:** Per-package plans (a) explode plan count for what is largely mechanical work. Single sweep (c) is unbisectable. Per-concern groups related changes so each plan has a single review focus.
**Note for planner:** Suggested plan layout (planner may refine):
1. `tier-classification` — add `protostar.tier` field to every package.json + rewrite AGENTS.md tier table (Q-04, Q-05, Q-06, Q-07).
2. `network-tier-no-fs-contracts` — add `no-fs.contract.test.ts` to `lmstudio-adapter` and `evaluation-runner` (Q-04a, Q-05).
3. `pure-tier-no-net-contracts` — add `no-net.contract.test.ts` to pure-tier packages (Q-06).
4. `workspace-conformance-gate` — single test in `admission-e2e` reading manifests + asserting tier rules + dep graph + tsconfig refs (Q-11, Q-12, Q-13).
5. `manifest-hygiene-sweep` — uniform `exports`, `sideEffects: false` on pure tier, `engines.node: '>=22'` on all published packages, knip extension for non-exported subpaths (Q-14, Q-15, Q-16).
6. `factory-cli-publish-wiring` — `publishConfig` + `bin` + post-install smoke for `@protostar/factory-cli` (Q-18).
7. `back-edge-audit` — grep imports for the three deferred back-edges; per-edge fix/flip/accept decision (Q-08, Q-09, Q-10).
**Status:** Decided.

## 2. Tier Classification

### Q-04 — `@protostar/lmstudio-adapter` tier
**Decision:** **Domain network tier — add `no-fs.contract.test.ts` + AGENTS row.** Treat identically to `dogpile-adapter` and `delivery-runtime`. AGENTS.md tier table grows a new row under "network-permitted, fs-forbidden".
**Rationale:** LM Studio is HTTP; the package is unambiguously network-tier. Missing contract is the most concrete boundary gap in the repo (operator's likely top reason for inserting this phase).
**Note for planner:** Heavy dep list (`execution`, `intent`, `authority`, `repo`, `artifacts`, `review`) is NOT audited in 10.1 (operator chose option a, not b). If grep during plan 7 (back-edge audit) shows `repo` is imported only for fs-tier types, flag for a separate phase — do NOT fix here.
**Status:** Decided.

### Q-05 — `@protostar/evaluation-runner` tier
**Decision:** **Domain network tier — add to AGENTS row under "network-permitted, fs-forbidden".** Coordinates judges via `dogpile-adapter`; transitively a network-using package. Existing `no-fs.contract.test.ts` is preserved; no new contract needed.
**Rationale:** Pure-orchestration framing (option c) understates the operational reality — every invocation drives a network-bound pile. Classification should reflect runtime behavior, not import-graph purity alone.
**Note for planner:** Already has `no-fs.contract.test.ts` — do NOT duplicate. Add only the AGENTS table row + `protostar.tier: "network"` field per Q-07.
**Status:** Decided.

### Q-06 — Tiers for `mechanical-checks`, `repair`, `fixtures`, `admission-e2e`
**Decision:** **All four = pure tier, with `admission-e2e` in a `test-only` sub-tier.** Introduce the `test-only` sub-tier explicitly so `admission-e2e`'s coupling to nearly every package is sanctioned, not accidental. Test-only packages can depend on any tier; nothing depends on them.
**Rationale:** The four packages have different shapes (mechanical-checks heavy, fixtures empty), but they share the pure invariant. The `test-only` distinction is load-bearing for `admission-e2e` because it's the one place a "depends on everything" pattern is correct.
**Note for planner:** AGENTS.md tier-table refresh adds: `pure: artifacts, authority, delivery, dogpile-types, evaluation, execution, fixtures, intent, mechanical-checks, paths-NO (paths is fs carve-out — keep separate), planning, policy, repair, review`; `test-only: admission-e2e`. Verify the list against `packages/` directory at planning time — the list above is from 2026-04-28 inventory.
**Status:** Decided.

### Q-07 — Declare tier in `package.json`
**Decision:** **Yes — add `protostar.tier: 'pure' | 'fs' | 'network' | 'orchestration' | 'test-only'` to every package.json.** Single source of truth; the workspace-wide conformance test (Q-11) reads this field and validates contract-test presence + dep direction.
**Rationale:** Manifests become the authoritative tier declaration; AGENTS.md prose is human documentation, not the load-bearing contract. Co-locates manifest hygiene with enforcement.
**Note for planner:** Custom field shape: `"protostar": { "tier": "pure" }`. The conformance test (Q-11) MUST refuse a package missing the field. Document the field schema in AGENTS.md alongside the tier table.
**Status:** Decided.

## 3. Dependency Back-Edges

### Q-08, Q-09, Q-10 — All back-edges deferred to planner
**Decision:** **Defer all three back-edges to a single back-edge-audit plan.** The planner's first task is to grep the actual import statements for:
- `intent → authority` (Q-08)
- `review → delivery` and `review → repair` (Q-09)
- `delivery-runtime → review` (Q-10)

For each edge, the planner picks one of: (a) flip/extract shared types, (b) accept and document in AGENTS.md as deliberate, or (c) treat as same fix (Q-10 explicitly tied to Q-09's resolution).

**Rationale:** The right call depends on what symbols are actually imported across each edge. Decision-without-data risks locking a fix the imports don't justify. Operator chose deferral on all three, signalling these aren't pre-decided in their head either.
**Note for planner:** Plan 7 (back-edge audit) deliverable is a one-screen summary per edge: imported symbols, recommended action (flip / extract / accept), and a concrete diff. If the audit reveals a structural problem that needs source moves, FLAG IT — that's out of 10.1 scope (Q-02 lock) and becomes a candidate phase 10.2 or a Phase 11 sub-task.
**Status:** Deferred to planner — explicit decision.

## 4. Enforcement Mechanism

### Q-11 — Workspace-wide tier-conformance test vs per-package contracts
**Decision:** **Both — workspace-wide is the gate, per-package contracts stay as defense-in-depth.** Workspace-wide test is authoritative; per-package contracts catch local edits without running the full e2e suite.
**Rationale:** Per-package contracts give tight feedback loops during local edits; workspace-wide test is the authoritative gate for "did the dep graph drift?" Both layers cost little; the redundancy is intentional defense-in-depth.
**Note for planner:** Workspace-wide test lives at `packages/admission-e2e/src/contracts/tier-conformance.contract.test.ts`. Per-package `no-fs.contract.test.ts` files stay in their respective `packages/*/src/`. Workspace-wide test SHOULD include a check that every package classified as `network` actually ships a `no-fs.contract.test.ts` (closes the "I forgot the local contract" gap).
**Status:** Decided.

### Q-12 — Dependency-direction enforcement tool
**Decision:** **Custom workspace-wide test (no new dev-dep).** Read each `package.json`'s `@protostar/*` deps, build the dep graph, assert no cycles, assert tier-respecting edges (e.g. `pure` may not depend on `network`).
**Rationale:** `dependency-cruiser` is mature but adds a dev-dep + JSON config — overkill for 20 packages and conflicts with the runtime-deps lock posture's spirit. Custom test matches the existing contract-test pattern, keeps the tooling story self-contained.
**Note for planner:** Test logic: read `pnpm-workspace.yaml`, glob `packages/*/package.json`, parse each, extract `@protostar/*` deps + `protostar.tier`, build graph, run cycle detection (Kahn's algorithm), assert tier-edge rules. Tier-edge rules: `test-only` may depend on anything; nothing may depend on `test-only`; `pure` may not depend on `network` or `fs`; `network` may not depend on `fs` (except types-only — flag if encountered); `orchestration` may depend on anything.
**Status:** Decided.

### Q-13 — tsconfig project-references alignment
**Decision:** **Add to the same workspace-wide conformance test.** Assert `package.json @protostar/* deps == tsconfig.json references` for every package.
**Rationale:** Drift between dep graph and TS project graph causes silent stale-dist bugs that don't surface until rebuild. Cheap to assert; lives in the same conformance test as Q-11/Q-12.
**Note for planner:** Allow tsconfig refs to be a SUPERSET of `@protostar/*` deps only when justified (e.g. type-only deps not yet declared). Default rule: exact equality. If superset is needed, add an allowlist comment in the conformance test with the package name + reason.
**Status:** Decided.

## 5. Manifest Hygiene

### Q-14 — `exports` field shape
**Decision:** **Uniform — every package documents allowed subpaths in its README; knip enforces "no import from a non-exported subpath".**
**Rationale:** Phase 10 Q-19 already ships knip; extending its config is cheaper than inventing new tooling. Per-package READMEs (Phase 10 Q-13) become the human-readable surface contract.
**Note for planner:** Knip config extension needs a check: scan `import` statements across the workspace, ensure every `@protostar/<pkg>/<subpath>` import resolves to an `exports` entry in that package's manifest. If knip can't natively express this, write a pre-knip script in `tools/check-subpath-exports.ts` (pure TS, no new dev-dep) and chain it into `pnpm verify`.
**Status:** Decided.

### Q-15 — `sideEffects: false` declarations
**Decision:** **Add `"sideEffects": false` to every pure-tier package; omit for fs/network tiers.**
**Rationale:** Mechanical decision once Q-04..Q-07 lands the tier classification. Pure packages are by definition side-effect-free at import time; declaring it helps downstream bundlers and documents the invariant.
**Note for planner:** Mechanical sweep: read `protostar.tier`, if `pure` (or `test-only`), add `"sideEffects": false`. Tie this assertion into the workspace-wide conformance test so future drift is caught.
**Status:** Decided.

### Q-16 — `engines` and `packageManager`
**Decision:** **Add `"engines": { "node": ">=22" }` to every published package; root keeps `packageManager`.**
**Rationale:** Public-package posture (Phase 10 Q-17) means npm consumers see `engines`; surfacing the Node-22 floor at install time prevents downstream surprise.
**Note for planner:** Mechanical. Source the version string from `process.engines.node` of root `package.json` if present; otherwise hard-code `">=22"`. Add to workspace-wide conformance test: every package with `private !== true` MUST declare `engines.node`.
**Status:** Decided.

### Q-17 — `peerDependencies` for cross-package types
**Decision:** **Keep all cross-deps as `dependencies: workspace:*`.** No peerDep migration in v0.1.
**Rationale:** No external consumers exist yet; speculative optimization. Lock simpler shape now; revisit only if a real consumer reports an install-footprint problem.
**Note for planner:** No-op for the manifest sweep. Document the decision in AGENTS.md prose so a future contributor doesn't try to "improve" by switching to peerDeps.
**Status:** Decided.

### Q-18 — `apps/factory-cli` published as `@protostar/factory-cli`
**Decision:** **Yes — confirm and wire per Phase 10 Q-17 note.** Add `publishConfig`, `bin`, validate the binary works post-install (e.g. `pnpm pack && npm install -g <tarball> && protostar-factory --help`).
**Rationale:** Phase 10 Q-17's note is currently aspirational; 10.1 turns it into shipped wiring.
**Note for planner:** `bin` entry: `"protostar-factory": "./dist/main.js"` (matches existing CLI shape per `STRUCTURE.md`). Post-install smoke test: a CI job (or a local script) that packs + installs + invokes `--help`. If the test is too fragile for CI in 10.1, ship the script + document the manual run instead of skipping the validation.
**Status:** Decided.
